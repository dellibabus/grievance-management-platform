Grievance Management Platform

REST API Documentation

Backend: Node.js + Express + TypeORM + PostgreSQL
Base URL (local): http://localhost:5000/api
Base URL (live): https://grievance-server-ua9w.onrender.com/api
Auth: JWT Bearer access token + httpOnly refresh-token cookie

Overview & Conventions

All authenticated endpoints require an Authorization: Bearer <accessToken> header. The refresh token is stored as an httpOnly cookie and exchanged via POST /api/auth/refresh.

Every response follows the shape { "success": boolean, ... }. Error responses follow:

{ "success": false, "message": "Description of the error" }

HTTP Status Codes

StatusMeaning
200 / 201Success / Resource created
400Validation error / bad request
401Missing or invalid access token
403Authenticated but lacks required role/permission
404Resource not found
500Unhandled server error

Roles & Permission Model

RBAC is implemented with two middlewares: authenticate (verifies JWT, attaches req.user) and authorize([permissions]) (checks req.user.permissions, sourced from roles.permissions jsonb array). Roles are no longer hard-coded — super_admin can create new roles and permissions dynamically via /api/roles.

Built-in RoleScope
super_adminFull access — all districts, user management, role management, audit logs
state_adminAll districts (read/manage complaints & dashboard), audit logs (read)
district_adminScoped to own district — complaints, users, dashboard
volunteerOnly complaints assigned to them; limited status transitions

1. Authentication — /api/auth

MethodPathAuthDescription
POST/registermanage_usersRegister a new staff user
POST/loginPublicAuthenticate, returns access token + sets refresh cookie
POST/refreshCookieExchange refresh cookie for a new access token
POST/logoutOptionalRevoke refresh token, clear cookie
GET/meAuthenticatedReturn current user's profile

POST /api/auth/login

// Request
{ "email": "admin@grievance.com", "password": "Admin@123" }

// Response 200
{
  "success": true,
  "accessToken": "<jwt>",
  "user": {
    "id": "uuid", "name": "...", "email": "...",
    "role": "super_admin",
    "permissions": ["view_dashboard","manage_users", "..."],
    "district": null
  }
}
Access tokens are short-lived (JWT_EXPIRES_IN, e.g. 15m). Refresh tokens are 7-day httpOnly cookies, persisted in the refresh_tokens table for server-side revocation.

ParamTypeNotes

POST /api/complaints (multipart/form-data)

Fields: title, description, citizen_name, citizen_phone, citizen_email?, category_id, district_id, mandal_id, village_id?, priority, attachments[]

// Response 201
{ "success": true, "ticket_number": "GRV-2026-00042", "id": "uuid", "message": "Grievance submitted successfully" }
Side effects: creates initial complaint_updates row, emits a Socket.io event to the district room, and notifies that district's admins.

PUT /api/complaints/:id

{ "status": "in_progress", "priority": "high" }
  • Setting status to resolved stamps resolved_at; any other value clears it.
  • district_admin restricted to own district. volunteer restricted to assigned tickets and cannot change priority.

POST /api/complaints/:id/assign

{ "assigned_to_id": "uuid", "notes": "Optional note" }

Assignee must be volunteer or district_admin in the same district. Auto-promotes status pending → assigned.

POST /api/complaints/:id/update

{ "comment": "Site visit completed.", "status": "in_progress" }

Volunteers may only set status to in_progress, resolved, closed, or rejected.

4. Roles & Permissions — /api/roles

Mutations restricted to super_admin. Powers the dynamic Roles & Permissions admin page — roles are no longer static seeder data.

MethodPathDescription
GET/permissionsList permission catalog
POST/permissionsCreate a new permission (name + description)
GET/List roles with permissions + user counts
POST/Create a new role
PUT/:idUpdate a role's permissions
DELETE/:idDelete a custom role
// POST /api/roles
{ "name": "Field Inspector", "permissions": ["view_complaints", "edit_complaint"] }
// -> normalized to "field_inspector"
  • Role name normalized: trimmed, lowercased, spaces → underscores.
  • DELETE blocked for built-in roles (super_admin, state_admin, district_admin, volunteer) or if any user still has that role.
  • Every mutation writes a CREATE_ROLE / UPDATE_ROLE / DELETE_ROLE / CREATE_PERMISSION entry to audit_logs.
